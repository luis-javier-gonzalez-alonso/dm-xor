dm-xor

Information-Theoretically Secure XOR-Split Device Mapper Target

Introduction

dm-xor is a stable, high-security Linux Device Mapper target that splits data streams on-the-fly across 2 to 8 physical blocks or virtual drives using an information-theoretically secure XOR-splitting secret sharing scheme.

Unlike traditional RAID mirroring, which replicates data identically on all disks, dm-xor generates independent random noise blocks on all disks except the final disk, which stores the cumulative XOR product.

Why use dm-xor? If an attacker steals $N-1$ of your disks, they gain absolutely zero information about your files. Reconstructing even a single bit of data is mathematically impossible without access to all $N$ disks.

Key Features

🔒 Absolute Security

Protects data against physical disk theft. If any single drive is missing, the remaining set is indistinguishable from random noise.

⚡ Non-blocking Workqueue

Write/encode operations are offloaded to an unbound, concurrent workqueue to prevent blocking the application thread.

🥷 Strict Metadata Hiding

Intentionally drops DISCARD and WRITE_ZEROES commands so free space is encrypted as random noise, preserving plausible deniability.

📦 Multi-Distro Support

Includes native packaging scripts and automated pipelines for Debian, RHEL/Fedora, and Arch Linux.

🛠️ Built-in DKMS Integration

Automatically rebuilds the kernel module whenever you update your system's Linux kernel.

Installation

Select your installation method or distribution format using the tabs below:

Install pre-built Debian Package

Download the .deb package from GitHub Releases and install it using apt or dpkg:

# Install dependencies
sudo apt update
sudo apt install dkms dh-dkms linux-headers-generic

# Install the package
sudo dpkg -i dm-xor-dkms_1.3.6-1_all.deb

Install pre-built RPM Package

Download the .rpm package from GitHub Releases and install it using dnf or rpm:

# Install dependencies and spec
sudo dnf install dkms kernel-devel

# Install the module package
sudo rpm -i dm-xor-dkms-1.3.6-1.noarch.rpm

Install Arch Package

Download the Arch package from GitHub Releases and install it using pacman:

# Install dependencies and package
sudo pacman -Syu dkms linux-headers
sudo pacman -U dm-xor-dkms-1.3.6-1-any.pkg.tar.zst

Source Build via DKMS (Cross-distro)

If your distribution is not listed above, compile and register the module from source using the Makefile helper:

# Clone repository
git clone https://github.com/luis-javier-gonzalez-alonso/dm-xor.git
cd dm-xor

# Install via DKMS
sudo make dkms-install

Usage Guide

Follow this step-by-step walkthrough to set up a secure virtual drive using dm-xor.

1. Create Virtual Backing Disks (For Testing)

If you don't have empty physical disks, you can simulate them using loopback files:

# Create 3 empty files of 100MB each
dd if=/dev/zero of=/tmp/disk1.img bs=1M count=100
dd if=/dev/zero of=/tmp/disk2.img bs=1M count=100
dd if=/dev/zero of=/tmp/disk3.img bs=1M count=100

# Associate files with loopback devices
sudo losetup /dev/loop10 /tmp/disk1.img
sudo losetup /dev/loop11 /tmp/disk2.img
sudo losetup /dev/loop12 /tmp/disk3.img

2. Construct the Target Type Table

The target parameter format is:

0 <size_in_sectors> xor <backing_device_1> <backing_device_2> ... <backing_device_N>

Calculate the sector size of the loopback drives (100MB = 204800 sectors) and load the module:

# Load kernel module
sudo modprobe dm-xor

# Create the virtual mapped drive
echo "0 204800 xor /dev/loop10 /dev/loop11 /dev/loop12" | sudo dmsetup create secure-xor

3. Initialization (Secure Setup)

To achieve plausible deniability, you must fill the physical drives with indistinguishable cryptographic noise before formatting. Wiping the virtual device with zeroes will cause `dm-xor` to securely encrypt them into random noise:

# Write encrypted zeroes across all underlying disks
sudo dd if=/dev/zero of=/dev/mapper/secure-xor bs=4M status=progress

4. Format and Mount the Drive

The new device is available at /dev/mapper/secure-xor. Treat it like any regular hard drive:

# Format with EXT4 filesystem
sudo mkfs.ext4 /dev/mapper/secure-xor

# Mount it
sudo mkdir -p /mnt/secure_data
sudo mount /dev/mapper/secure-xor /mnt/secure_data

# Create a test file
echo "Confidential Data" | sudo tee /mnt/secure_data/secrets.txt

5. Best Practices: Secure Deletion

Because dm-xor intentionally drops DISCARD (TRIM) operations to prevent metadata leakage, deleting a file via standard rm leaves the encrypted plaintext on disk until overwritten.

To permanently destroy sensitive files, securely overwrite them with zeroes or random data using the shred utility before deletion:

shred -u /path/to/sensitive/file

6. Verify Security

Unmount and remove the target to sync data to backing files, and inspect the individual files. You'll notice they look like random static garbage:

# Unmount and clean up target
sudo umount /mnt/secure_data
sudo dmsetup remove secure-xor

# View the raw contents of disk1 (random noise)
hexdump -C /tmp/disk1.img | head -n 10

Technical Design

Secret Sharing Mathematics

Let $D_i$ be the $i$-th backing device (where $0 \le i < N$, and $N$ is the total device count). When writing a data block $S$ to the virtual device, the encoding routine computes:

$$D_i = \text{random\_noise} \quad \text{for } 0 \le i < N-1$$

$$D_{N-1} = S \oplus D_0 \oplus D_1 \oplus \dots \oplus D_{N-2}$$

When reading, the data block $S$ is recovered instantaneously by XORing all segments together:

$$S = D_0 \oplus D_1 \oplus \dots \oplus D_{N-1}$$

Unbound Workqueue Pipeline

Performing XOR operations and invoking random generation blocks on high-bandwidth read/write pipelines can consume significant CPU cycles. If run on the block layer mapping threads, it blocks caller applications.

To solve this, dm-xor offloads computational processes to a custom dedicated Linux workqueue (dm_xor):

  • Write encoding: When a write bio arrives in xor_map(), bounce pages are allocated, clone bios are configured, and the encryption task is queued to the workqueue. The block layer caller thread returns immediately with DM_MAPIO_SUBMITTED.
  • Read decoding: When read clones complete in xor_end_io(), decoding requires kernel virtual address local page mappings (kmap_local_page). The callback schedules the decode task onto the workqueue, freeing the block layer softirq context.

Zero Post-Map State Reliance

A common source of instability in custom Device Mapper modules is relying on bio iterator structures that change asynchronously during request lifecycles.

dm-xor takes a snapshot of all segment lengths, offsets, and page pointers into direct arrays inside a local tracker object (struct xor_io_tracker) at the moment of entry. All operations (encode, clone submission, decode, and page frees) iterate this static snapshot, eliminating concurrency and race hazards.

Suspend & Resume

Unlike block devices that implement complex suspend state machines, dm-xor leverages the Linux Device Mapper framework's built-in target suspend infrastructure. During table suspension, the block layer blocks incoming requests and automatically flushes the workqueue and waits for all in-flight bio clones to complete. This ensures data consistency without adding custom state overhead to the driver code.
